# 7. How teams do it in production

Every large search system converges on the same skeleton: understand the query,
retrieve with a lexical arm and an embedding arm, rank the survivors with a
learning-to-rank model trained on fused human judgments plus debiased engagement.
What actually differs is two decisions: **which arm dominates retrieval**, and
**where the labeling budget goes**. The architecture everyone shares; the leverage
is in the arm selection and the label pipeline.

## Where the real designs diverge

| System | Query understanding | Retrieval | Ranking model | Label source | Why this shape |
|---|---|---|---|---|---|
| Amazon Music (bandit) | structured parse + query-to-entity memory | contextual bandit selects from BM25, BERT, SPLADE, field-match | neural LTR on click/like/play | logged engagement + bandit exploration | right candidate never enters if retrieval is static; bandit learns which arm fits each query context |
| LinkedIn post search | IQL translation to Galene | hybrid inverted index | GBDT first pass, neural second pass, diversity re-ranker | crowdsourced ratings + engagement | cheap GBDT casts a wide recall net; neural pass adds precision within budget |
| Pinterest SearchSage | DistilBERT query encoder | ANN over frozen PinSage embeddings (HNSW) | multi-objective relevance and engagement | saves + 35s long clicks | freezing the document tower lets all 15+ downstream systems reuse one index without retraining |
| Instacart hybrid | category and attribute parse | text (Postgres GIN + ts_rank) + semantic (MiniLM FAISS), adaptive by query entropy | two-stage ranking | engagement + conversion logs | grocery catalog has crisp categories and conversion as a clean downstream label; entropy-adaptive recall avoids over-fetching on broad queries |
| Instacart intent engine | LLM intent + category mapping via LoRA Llama-3-8B student | feeds downstream retrieval | downstream LTR | LLM-labeled intents + engagement | broad and ambiguous queries need LLM-scale understanding; teacher-student split keeps tail-query latency under 300ms |
| Yelp business matching | input normalization | Elasticsearch candidate retrieval | pointwise regression LTR | manually labeled gold dataset | task is match-or-not per candidate, not open-ended ranking; pointwise regression is the right framing |
| GetYourGuide activity search | n/a (activities have no free-text query) | n/a | LTR with discounted-ranking-impressions feature | bookings + position-discounted impressions | position bias is baked into a single feature rather than a separate debiasing model; Tecton + Redis + FastAPI stack hits p99 under 80ms |
| Booking.com property search | n/a | per-shard availability search | multi-stage: per-shard then post-merge | real-time price + availability + user behavior | volatile price/availability features must be stream-computed; static fallback score prevents blank results on model failure |
| Spotify podcast search | shared-weight dual encoder (USE-CMLM) | Vespa ANN + Elasticsearch (fused) | final reranker (cosine as one feature) | successful searches + BART-synthetic pairs + reformulations | semantic search is an additional arm beside Elasticsearch, not a replacement; exact-match queries still need the keyword arm |
| Google DCN V2 | n/a (ranking model) | n/a | deep and cross network with explicit low-rank feature crosses | web-scale clicks | explicit feature crosses that MLP cannot learn implicitly; low-rank factorization keeps the cross matrix affordable |

The dividing line is simple: **which arm dominates depends on the query mix and
catalog type; where the labeling budget goes depends on how much of the signal
is biased clicks versus trustworthy conversions or judgments.** A complete answer
picks a point on both axes and justifies it from the problem statement.

## The systems (first-party write-ups)

- **Amazon** [From structured search to learning-to-rank-and-retrieve](https://www.amazon.science/blog/from-structured-search-to-learning-to-rank-and-retrieve): a contextual bandit selects which retrieval strategies (BM25, BERT, SPLADE, field-match, entity memory) to fire per query; a neural LTR model reranks the union. The key insight: a perfect ranker is useless if the right candidate never enters the set, so retrieval-strategy selection must itself be learned. *(product design)*
- **LinkedIn** [Improving Post Search at LinkedIn](https://www.linkedin.com/blog/engineering/search/improving-post-search-at-linkedin): multi-stage funnel with an IQL translation layer, a GBDT first-pass ranker covering five aspects (relevance, quality, personalization, engagement, recency), a neural second-pass ranker in the federation layer, and a diversity re-ranker. *(product design)*
- **Pinterest** [SearchSage: learning search query representations](https://medium.com/pinterest-engineering/searchsage-learning-search-query-representations-at-pinterest-654f2bb887fc): a small multilingual DistilBERT query tower trained to embed into the frozen PinSage Pin-embedding space, powering ANN retrieval and ranking relevance across 15+ systems. *(deployment)*
- **Instacart** [Optimizing search relevance using hybrid retrieval](https://tech.instacart.com/optimizing-search-relevance-at-instacart-using-hybrid-retrieval-88cb579b959c): Postgres GIN text retrieval fused with MiniLM-L3-v2 FAISS semantic retrieval via RRF or convex combination, with query-entropy-adaptive recall thresholds. *(deployment)*
- **Instacart** [Building the Intent Engine: query understanding with LLMs](https://company.instacart.com/tech-innovation/building-the-intent-engine-how-instacart-is-revamping-query-understanding-with-llms): LLM-based intent classification, category mapping, and query rewrites using a LoRA Llama-3-8B student distilled from an offline RAG teacher; meaningfully reduced poor-tail-result complaints in internal evaluation. *(product design)*
- **Yelp** [Learning to Rank for Business Matching](https://engineeringblog.yelp.com/2014/12/learning-to-rank-for-business-matching.html): moving from hand-tuned weights to a pointwise regression LTR over Elasticsearch component scores and a manually labeled gold dataset, with a meaningful F1 improvement over the hand-tuned baseline. *(product design)*
- **Wayfair** [WANDS: a public e-commerce product-search relevance dataset](https://www.aboutwayfair.com/careers/tech-blog/wayfair-releases-wands-the-largest-and-richest-publicly-available-dataset-for-e-commerce-product-search-relevance): 480 queries, 42,994 products, 233,000 human-judged query-product relevance labels. The reference for what a well-constructed human-judgment evaluation set looks like. *(eval bar)*
- **GetYourGuide** [Powering Millions of Real-Time Rankings with Production AI](https://www.getyourguide.careers/posts/powering-millions-of-real-time-rankings-with-production-ai): 30M+ daily ranking predictions under 80ms end-to-end. Tecton feature store with Stream and On-Demand Feature Views, Airflow daily training pipeline, FastAPI + Kubernetes serving, Redis online store at p99 7ms, and Arize drift and NDCG monitoring. Point-in-time join is the critical correctness step. *(deployment)*
- **Booking.com** [The Engineering Behind High-Performance Ranking Platform](https://medium.com/booking-com-development/the-engineering-behind-booking-coms-ranking-platform-a-system-overview-2fb222003ca6): multi-stage ranking with per-shard scoring and a post-merge pass; tiered feature computation (static, slow-changing, real-time stream); static fallback score on inference failure. p999 sub-second target. *(product design)*
- **Shopify** [How Shopify improved consumer search intent with real-time ML](https://shopify.engineering/how-shopify-improved-consumer-search-intent-with-real-time-ml): real-time streaming embedding pipeline on Google Cloud Dataflow sustaining 2,500 embeddings per second; batching host-to-device transfers cut CPU-GPU bottleneck; thread-concurrency tuning cut memory 2.6x. *(deployment)*
- **Spotify** [Introducing Natural Language Search for Podcast Episodes](https://engineering.atspotify.com/2022/03/introducing-natural-language-search-for-podcast-episodes/): shared-weight USE-CMLM dual encoder, episode embeddings precomputed in Vespa, query vectors on Vertex AI with caching. Semantic retrieval is an additional arm beside Elasticsearch; cosine similarity is one feature in the final reranker, not the entire ranking signal. *(product design)*
- **Google (Wang et al.)** [DCN V2: Improved Deep and Cross Network](https://arxiv.org/abs/2008.13535): explicit, efficient feature crosses in a ranking model at web scale; low-rank mixture variant keeps the cross matrix affordable. *(ranking model)*
- **Google (Cheng et al.)** [Wide and Deep Learning](https://arxiv.org/abs/1606.07792): jointly trains a linear model over crossed categorical features (memorization) with a deep MLP over learned embeddings (generalization). *(ranking model)*
- **Burges** [From RankNet to LambdaRank to LambdaMART: An Overview](https://www.microsoft.com/en-us/research/publication/from-ranknet-to-lambdarank-to-lambdamart-an-overview/): the canonical learning-to-rank reference, walking from a pairwise RankNet loss to NDCG-weighted LambdaRank gradients to the LambdaMART tree ensemble. The clearest single source on why ranking losses are pairwise and listwise rather than pointwise. *(learning-to-rank)*
